## Project Overview

**LongcellPre** is an R package for preprocessing Nanopore long-read single-cell RNA-seq data from 10X Chromium experiments. It takes FASTQ input, performs barcode/UMI extraction, genome alignment (via external `minimap2`/`samtools`/`bedtools`), isoform extraction, UMI deduplication, and produces cell-by-isoform expression matrices. The heavy computation is delegated to the companion C++ package [`Longcellsrc`](https://github.com/yuntianf/Longcellsrc) via Rcpp/RcppEigen.

- **Language:** R (100%). No Python, no shell scripts other than `exec/RunLongcellPre.R`.
- **R version:** >= 4.0 (implied by dependencies).
- **Build system:** Standard R package using `devtools` + `roxygen2`.


## Code Style Guidelines

### Indentation & Formatting

- **2 spaces** per indent level. Never use tabs.
- Opening brace `{` on the same line as the statement (K&R style).
- Closing `}` on its own line.
- Long argument lists: break across lines with each argument aligned under the first, or one per line with 2-space indent from the function name.
- One blank line between top-level function definitions.
- Files must end with a trailing newline.

### Assignment

- Prefer `=` for most assignments (dominant style, ~80% of codebase).
- `<-` is acceptable, especially for conventional R patterns (`start_time <- Sys.time()`) or when following Bioconductor style in a block.
- Be consistent within a function — do not mix `=` and `<-` freely.

### Naming Conventions

| Entity | Convention | Examples |
|---|---|---|
| Internal functions | `snake_case` | `gene_reads_extraction`, `umi_cluster` |
| Exported API functions | `PascalCase` or `camelCase` | `RunLongcellPre`, `BarcodeMatch`, `extractTagBc` |
| Variables / parameters | `snake_case` | `gene_bed`, `barcode_len`, `splice_site_thresh` |
| Files | `snake_case` | `reads_extraction.R`, `umi_cluster.R` |
| Constants / abbreviations | Short uppercase ok | `DT`, `CNT`, `bc`, `uc` |

### Imports & Namespace Usage

- **Never edit `NAMESPACE` by hand.** It is entirely auto-generated by `roxygen2`. Run `devtools::document()` to regenerate it.
- Every function that uses an external package function **must** carry an `@importFrom package function` roxygen2 tag at the top of its documentation block.
- Use **explicit namespace qualification** (`package::function()`) for any call that could be masked by another package, particularly:
  - `dplyr::select()`, `dplyr::filter()`, `dplyr::rename()`
  - `Matrix::sparseMatrix()`, `Matrix::summary()`
  - `Longcellsrc::*`
  - `valr::bed_merge()`
  - `peakRAM::peakRAM()`
- The pipe `%>%` must be declared with `@importFrom magrittr "%>%"`.

### Roxygen2 Documentation

Every function — including non-exported helpers — must have a roxygen2 block:

```r
#' @title function_name
#'
#' @description One-sentence description.
#' @details Optional longer explanation. Use \enumerate{} or \itemize{} for
#'   lists, \code{} for inline code.
#'
#' @param param_name Description of the parameter and its expected type/range.
#' @param param2,param3 Shared description for multiple related params.
#' @inheritParams other_function
#'
#' @importFrom dplyr group_by summarise
#' @importFrom magrittr "%>%"
#'
#' @return Description of the returned object (class, structure).
#' @export   # only for public API functions
```

- Add `@export` **only** to functions intended as public API (roughly 15 out of ~80 in this package).
- `@details` is optional for short/obvious functions.

### Data Manipulation Style

- **Legacy/simple code:** `dplyr` pipe chains with `%>%`. Always pass `.groups = "drop"` to `dplyr::summarise()`.
- **New performance-critical code:** Use `data.table` idioms — `:=`, `setorder()`, `setnames()`, `setkey()`, `rbindlist()`, `[, by = .()]`. Prefer `data.table` over `dplyr` for functions operating on large per-gene or per-cell data frames.
- Avoid `apply`/`lapply` loops over large data when a vectorized or `data.table` solution exists.
- Parallelism: use `future.apply::future_lapply()` with a `future` plan configured by the caller; do not set the plan inside library functions.

### Error Handling

- `stop("message")` — unrecoverable errors (bad arguments, missing required files).
- `warning("message")` — recoverable issues (e.g., annotation already exists).
- `message("message")` — informational output to `stderr` (installation steps, resource loading).
- `cat("...\n")` — pipeline progress status to `stdout`.
- Wrap external tool calls and per-element iterations in `tryCatch`:
  ```r
  tryCatch({
    result = some_external_call(x)
  }, error = function(e) {
    warning(sprintf("Failed for '%s': %s", x, e$message))
    NULL
  })
  ```
- Use `suppressWarnings()` only for known-harmless coercions (e.g., `as.numeric` on strings).

### Type Checking

R has no static types. Use manual guards:
```r
if (is.null(x)) stop("x must not be NULL")
if (!is.data.frame(data)) stop("data must be a data.frame")
```
Do not use third-party assertion libraries (`checkmate`, `assertthat`) unless they are added to `DESCRIPTION`.

### Parallelism

- Entry-point functions accept a `cores` parameter and call `coreDetect(cores)` (see `parallel.R`) to resolve the actual core count.
- Use `genes_distribute()` / `dataSplit()` to partition work across cores.
- Parallelism is implemented via `future` + `future.apply`; the `future` plan is set by the top-level `RunLongcellPre` or the user, never inside a helper function.

## Common Pitfalls

- **Do not edit `NAMESPACE` or `man/*.Rd` directly.** Always regenerate with `devtools::document()`.
- **Install `Longcellsrc` before attempting to load or check `LongcellPre`.** It is a hard `LinkingTo` dependency.
- **External binaries** (`minimap2`, `samtools`, `bedtools`) must be on `PATH` at runtime; the package does not bundle them.
- **Bioconductor packages** (`GenomicAlignments`, `GenomicFeatures`, `Rsamtools`, `BSgenome`, etc.) must be installed via `BiocManager::install()`, not `install.packages()`.
- When adding a new exported function, add `@export` to its roxygen block **and** re-run `devtools::document()` before testing — otherwise R will not find the function.
- Avoid leaving large blocks of commented-out code; remove dead code or mark it with a `# TODO:` comment explaining why it is retained.
